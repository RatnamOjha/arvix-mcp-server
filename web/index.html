MCP Tools
Six tools.
One research assistant.
01
search_papers
query, max_results?, category?

Live ArXiv semantic search. Returns ranked results with titles, authors, abstracts, categories, and PDF links.

02
fetch_paper
arxiv_id, add_to_reading_list?

Downloads the full PDF directly from ArXiv's CDN (no rate limiting), extracts text with pymupdf, chunks at 500 words, indexes into BM25.

03
query_library
question, top_k?, arxiv_id_filter?

Full vectorless RAG: query expansion → multi-query BM25 → RRF fusion → sentence-level compression → cited context. Filter to one paper with arxiv_id_filter.

04

arxiv_id

Pulls intro + methods + conclusion chunks and asks Claude for a structured summary: problem, contribution, results, limitations.

05
list_library

Returns all papers indexed in your local BM25 store with titles, authors, categories, and index dates.

06
get_reading_list
—

Your saved reading list: metadata, timestamps, read/unread status, personal notes — persisted to JSON.

How it works without Pro
Two ways to use this

Without Claude Pro — the full ML pipeline runs locally from your terminal. Use test_local.py to search ArXiv, fetch papers, run RAG queries, and inspect your library. Use test_mcp.py to test the actual MCP protocol — byte-for-byte identical to what Claude Desktop sends.

With Claude Pro — connect the server to Claude Desktop and research in natural language. Claude calls the 6 tools automatically based on what you ask.

Finding your paper ID

Every ArXiv URL contains the ID at the end:

https://arxiv.org/abs/2301.07041

That number is all you need. Version suffixes like v2 are stripped automatically.

Clone and install
git clone https://github.com/RatnamOjha/arvix-mcp-server cd arvix-mcp-server python3 -m venv .venv && source .venv/bin/activate pip install -e .
Anaconda users: use the full venv path to avoid conflicts — /path/to/project/.venv/bin/python3 test_local.py

Search ArXiv live
python3 test_local.py --search
Queries the live ArXiv API and returns ranked results with titles, authors, and IDs. No paper needed yet.
3
Fetch a paper and run RAG
python3 test_local.py --fetch --arxiv-id 2005.11401 python3 test_local.py --query --question "what retrieval method does RAG use?" --full
Fetches the original RAG paper, indexes it, then runs the full BM25 + RRF + compression pipeline. --full shows the complete untruncated output.
4
Run everything at once
python3 test_local.py
End-to-end demo: search → fetch → query → library. Good for a first run.
1
Fetch and index your paper
python3 test_local.py --fetch --arxiv-id 2301.07041
Downloads the PDF directly from ArXiv's CDN — no API rate limiting. Extracts text with pymupdf, builds BM25 index. Takes ~15 seconds.
2
Ask questions — scoped to your paper
python3 test_local.py --query --paper 2301.07041 --question "what problem does this solve?" --full python3 test_local.py --query --paper 2301.07041 --question "what are the limitations?" --full python3 test_local.py --query --paper 2301.07041 --question "what datasets did they use?" --full
--paper restricts BM25 search to that one paper only — no cross-contamination from other indexed papers. --full shows untruncated context.
3
Build a multi-paper library
python3 test_local.py --fetch --arxiv-id 2301.07041 python3 test_local.py --fetch --arxiv-id 2305.10601 python3 test_local.py --fetch --arxiv-id 2005.11401 python3 test_local.py --query --question "how do these papers approach retrieval differently?" --full
Papers persist at ~/.arxiv-mcp/ between runs. Query across the whole library by omitting --paper. BM25 scales linearly — add as many as you want.
4
Check your library and reading list
python3 test_local.py --library
Lists all indexed papers with titles, authors, chunk counts, and reading list status.
why
What test_mcp.py actually does

It spawns the MCP server as a subprocess and sends real JSON-RPC messages over stdin/stdout — byte-for-byte identical to what Claude Desktop sends. If this passes, the Claude Desktop integration will work. It's the closest you can get to testing the full MCP stack without a Pro subscription.

1
Test all tools at once
python3 test_mcp.py
Runs the full initialize handshake, lists tools, and calls list_library and get_reading_list.
2
Test specific tools
python3 test_mcp.py --tool search_papers python3 test_mcp.py --tool fetch_paper --arxiv-id 2005.11401 python3 test_mcp.py --tool query_library --question "what is BM25?" --paper 2005.11401 python3 test_mcp.py --tool list_library
Each call goes through the full MCP protocol stack — tool registration, JSON-RPC serialization, and response parsing.
Edge cases & fixes
Problems we hit.
How we solved them.
429
ArXiv API rate limiting

The ArXiv metadata API limits ~1 request/3 seconds. Burst fetching triggers HTTP 429.

# Old: blocked on API rate limit fetch metadata via API → get PDF URL → download # New: guaranteed, no rate limit download PDF directly → https://arxiv.org/pdf/{id} fetch metadata separately with graceful fallback
PDF hits ArXiv's CDN directly — no rate limit ever. If metadata API is unavailable, placeholder values are used and the paper is still fully indexed and queryable.
PDF
Garbled text from LaTeX PDFs

pdfplumber misreads LaTeX fonts — producing b e a h A b r ξ o a s α instead of sentences.

pip install pymupdf # primary extractor # pdfplumber remains as automatic fallback # post-processing drops lines with avg token length < 1.8 chars
pymupdf reconstructs reading order from glyph positions and handles multi-column LaTeX layouts correctly.
ENV
Anaconda overriding the virtualenv

Both (base) and (.venv) in your prompt means conda is winning.

conda deactivate source .venv/bin/activate which python3 # must show .venv path # Or bypass entirely /full/path/.venv/bin/python3 test_local.py --fetch --arxiv-id 2005.11401
Always use the full absolute .venv path in your Claude Desktop config — avoids the conflict regardless of conda.
OUT
Truncated terminal output

Default view shows 600 characters. Long papers or multi-source queries get cut off.

python3 test_local.py --query --paper 2507.07171 \ --question "what methods do they use?" --full
--full removes truncation entirely. Always use it — truncated context is never useful.
Architecture
How it works
New in v2
Vectorless RAG

Traditional RAG embeds every chunk with a neural model and retrieves by cosine similarity. Vectorless RAG replaces the embedding step with BM25 keyword retrieval — the same algorithm that powers Elasticsearch and academic search engines. It excels at exact technical terms like LoRA, RLHF, KV-cache that dense vectors often miss. Multi-query results are fused with Reciprocal Rank Fusion (RRF), then each chunk is compressed to only the relevant sentences by a sentence-level BM25 pass.

step 01
Query expansion
Rewrites query into 3 keyword variants to maximize BM25 recall
step 02
Multi-query BM25
Okapi BM25 scores all chunks for each variant independently
step 03
RRF fusion
score = Σ 1/(60 + rank) — combines 3 ranked lists without score normalization
step 04
Contextual compression
Sentence-level BM25 extracts only the 2–3 sentences that answer the question
01 —
Claude calls a tool

Claude Desktop connects over stdio via MCP. Research questions route to the appropriate tool as structured JSON.

02 —
PDF downloaded directly

PDFs fetched from https://arxiv.org/pdf/{id} — ArXiv's CDN with no rate limiting. pymupdf extracts text, handles LaTeX multi-column layouts.

03 —
Chunking → BM25 index

500-word chunks with 50-word overlap. Each chunk added to Okapi BM25 instantly — no encode() call, no model.

04 —
Multi-query retrieval + RRF

Three BM25 searches fused by RRF. Optional arxiv_id_filter restricts search to one paper.

05 —
Cited context returned

Compressed chunks formatted as [Source N] citations in structured JSON. Claude generates a grounded, cited answer.
